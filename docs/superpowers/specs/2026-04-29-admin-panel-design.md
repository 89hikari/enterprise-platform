# Admin Panel вЂ” Company Structure Management

**Date:** 2026-04-29
**Status:** Approved

---

## Overview

An ongoing admin panel for superadmins and admins to manage the company's organizational structure: edit the organization, create and arrange departments hierarchically, and invite users (creating them simultaneously in Keycloak and the application database).

---

## Backend

### KeycloakAdminService

**Location:** `apps/api/src/modules/keycloak-admin/keycloak-admin.service.ts`

A new NestJS injectable service that wraps Keycloak's Admin REST API.

**Authentication:** Obtains an admin token via client-credentials grant against the `master` realm using `KEYCLOAK_ADMIN` / `KEYCLOAK_ADMIN_PASSWORD` from env. Token is cached in memory and refreshed automatically when expired (checked before each request).

**Public methods:**

| Method | Description |
|---|---|
| `createUser(data)` | POST to `/auth/admin/realms/enterprise/users`. Sets `UPDATE_PASSWORD` as a required action so the user must change their password on first login. Assigns the provided realm roles. Returns the Keycloak user UUID extracted from the `Location` response header. |
| `deleteUser(sub)` | DELETE to `/auth/admin/realms/enterprise/users/:sub`. Used for rollback only. |

**Module:** `KeycloakAdminModule` exports `KeycloakAdminService` and is imported by `UsersModule`.

---

### UsersService / CreateUserDto changes

`CreateUserDto` changes:
- Remove `keycloakSub` (no longer supplied by caller вЂ” generated by `KeycloakAdminService`)
- Add `password: string` (required, min 8 chars) вЂ” the temporary password the admin sets

`UsersService.create()` new flow:
1. Check email uniqueness in DB (existing check)
2. Call `KeycloakAdminService.createUser({ email, firstName, lastName, password, roles: [dto.role] })` в†’ receive `keycloakSub`
3. `prisma.user.create({ id: keycloakSub, keycloakSub, ...rest })`
4. If step 3 throws: call `KeycloakAdminService.deleteUser(keycloakSub)` to roll back, then re-throw

No changes to any other endpoint or service.

## Frontend

### Sidebar

Add an **Admin** entry in `Sidebar.tsx`, rendered only when `auth.user` token contains role `superadmin` or `admin`. Positioned below existing nav items.

---

### Route

`/admin` вЂ” new page at `apps/web/src/app/(app)/admin/page.tsx`

Three tabs rendered client-side: **Organization**, **Departments**, **Users**.

---

### Organization Tab

- Displays current organization name and slug (fetched from `GET /api/organizations`)
- Inline edit for name: click pencil icon в†’ text input в†’ save via `PATCH /api/organizations/:id`
- Slug is read-only (changing it would break existing references)

---

### Departments Tab

Hierarchical tree built from the flat department list returned by `GET /api/departments` (which includes `parentDepartmentId`). The tree is constructed client-side.

**Each node:**
- Department name (click to edit inline, blur to save via `PATCH`)
- **+ Child** button вЂ” opens a small inline input to name and create a child department via `POST /api/departments` with `parentDepartmentId` set
- **Г— Delete** button вЂ” calls `DELETE /api/departments/:id`; the API already returns 400 if the department has members, which is surfaced as an error toast

**Root level:**
- **+ Add department** button at the top creates a top-level department (no `parentDepartmentId`)

Tree is re-fetched after every mutation.

---

### Users Tab

**List:** Paginated table with columns: Avatar, Name, Email, Role (badge), Department, Status (Active/Inactive). Row action: **Deactivate** (calls `PATCH /api/users/:id/deactivate`), shown only for active users.

**"+ Add User" modal form fields:**

| Field | Type | Notes |
|---|---|---|
| First name | text | required |
| Last name | text | required |
| Email | email | required |
| Role | select | employee, manager, hr_manager, admin, superadmin |
| Department | select | populated from department list, optional |
| Position | text | optional |
| Temporary password | password | required, min 8 chars; user must change on first login |

On submit: `POST /api/users`. On success: close modal, invalidate users query, show success toast. On error: show error message inline.

---

## Access Control

The `/admin` route and the sidebar link are only shown to users with role `superadmin` or `admin`. API endpoints already enforce this via `@Roles` guards on the relevant controllers.

---

## Environment Variables

`KEYCLOAK_ADMIN` and `KEYCLOAK_ADMIN_PASSWORD` are already set in `infra/docker-compose.dev.yml` (values: `admin` / `admin`). They must be added to `apps/api/.env` and `.env.dev` so the API process can read them at runtime:

```
KEYCLOAK_ADMIN=admin
KEYCLOAK_ADMIN_PASSWORD=admin
```

Also add them to `app.config.ts` under the `keycloak` block.

---

## Minor API Fix Required

`PATCH /api/organizations/:id` currently has no `@Roles` guard. Add `@Roles('superadmin', 'admin')` to that handler before the admin panel ships.

---

## Out of Scope

- Bulk CSV import of users or departments
- Editing Keycloak user details after creation (password reset, etc.)
- Organization creation (the organization already exists from seed)
- Deleting users (deactivation is sufficient)
